# ADR-004: 자동 확정 기반 예약 플로우

## 상태

Accepted (2026-03-28)

---

## 배경

예약 생성 시 상태 전이 방식에는 크게 두 가지 접근이 있다.

1. 즉시 확정 (CONFIRMED): 재고 차감 성공과 동시에 예약이 확정 상태로 생성된다.
2. PENDING 경유: 예약이 먼저 대기 상태로 생성되고, 이후 파트너 확인 또는 내부 처리 과정을 거쳐 확정된다.

PENDING 경유 방식이 적합한 경우:

- 숙소 파트너가 각 예약을 수동으로 승인하는 "요청형 예약" 모델 (ex. Booking.com 리서치에서 확인됨)
- 외부 채널 OTA와 API 통신을 통해 재고를 재검증해야 하는 경우
- 이 경우 `PENDING 생성` → `외부 확인` → `CONFIRMED 또는 REJECTED` 흐름이 필요하다.

반면 본 플랫폼은 자동 확정이 적합하다.

- 자체 재고(`inventory` 테이블)를 직접 관리한다.
- 재고 차감이 DB 트랜잭션 내에서 원자적으로 처리된다.
- 재고 차감에 성공했다면 추가적인 외부 확인 없이 예약의 유효함이 보장된다.

---

## 결정

- 자동 확정 플로우를 채택한다.

```
[고객 예약 요청]
       ↓
[재고 확인 + 차감 (SELECT FOR UPDATE)]
       ↓ 성공
[Reservation 생성 — 상태: CONFIRMED]
       ↓
[ApplicationEvent 발행 (트랜잭션 외부)]
       ↓
[채널 매니저 동기화 알림 (비동기, DESIGN-ONLY)]
```

재고 부족 시에는 예약 자체가 생성되지 않고 즉시 오류 응답을 반환한다.

---

## 대안

- Option A: PENDING → 파트너 수동 확정
  - 장점
    - 파트너가 예약을 검토할 기회가 있다.
    - 특수한 날짜(특가 행사 등)에 유연하게 대응 가능하다.
  - 단점
    - 고객 UX 저하: 파트너 응답을 기다리는 동안 고객은 예약 확정 여부를 알 수 없다. 
    - 구현 복잡도: 만료 타이머, 파트너 알림, 상태 전이 처리 등 부가 로직이 상당하다.

---

## 긍정적 결과

- 고객은 예약 버튼을 누른 즉시 확정 이메일을 받는다.
  - UX가 단순하고 명확하다.
- 예약 상태 관리 로직이 단순하다.
  - `CONFIRMED` → `CANCELLED` 단방향 전이만 처리한다.
- PENDING 만료 타이머, 파트너 알림 큐 등 부가 인프라가 불필요하다.


---

## 후속 조치

- 파트너 수동 확정 모드를 파트너별 설정으로 활성화할 수 있는 구조 설계 (DESIGN-ONLY)
  - `partner.confirmation_mode = AUTO | MANUAL` 필드 추가
  - MANUAL 모드에서는 PENDING 상태 생성 + 파트너 알림 + 만료 처리 필요
- 파트너 귀책 취소에 대한 고객 보상 정책 구현
