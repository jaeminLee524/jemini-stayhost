# ADR-002: 비관적 락 기반 예약 동시성 처리

## 상태

Accepted (2026-03-28)

---

## 배경

OTA 플랫폼에서 예약 생성은 반드시 재고 차감과 원자적으로 처리되어야 한다. 특히 다음 시나리오가 핵심 문제다.

- 마지막 1개 객실에 대한 동시성 처리
- 멀티 나이트 예약: 연박의 경우 모든 날짜 재고가 동시에 차감되어야 한다.


---

## 결정

- 해당 결정은 모두 단일 서버를 기준으로 결정했다.
- 분산 서버의 경우 Redis 분산 락과 각 로컬 캐시, 그리고 로컬 캐시 동기화 작업 등 여러가지 고민을 하여 진행해야할 것 같다.
- (상세: [04-concurrency.md](../04-concurrency.md) 7.3절)

```
- 1차: 애플리케이션 레벨에서는 Caffeine CAS(AtomicInteger)를 1차 필터로 사용하여 DB에 도달하기 전에 매진 요청을 걸러낸다.
- 2차: MySQL `SELECT ... FOR UPDATE` (비관적 락) 을 채택하여 DB 레벨에서 재고 초과 예약(overbooking)을 원천 차단한다.
```

- 예약 생성 트랜잭션 시작 시 대상 날짜의 `inventory` 레코드 전체를 `SELECT ... FOR UPDATE`로 획득한다
- 락을 획득한 트랜잭션만 재고 수량을 읽고 차감할 수 있으며, 다른 트랜잭션은 락이 해제될 때까지 대기한다
- 데드락 방지를 위해 멀티 나이트 예약 시 항상 `date ASC` 순서로 락을 획득한다

```sql
SELECT * 
  FROM inventory
 WHERE room_type_id = :roomTypeId
   AND date BETWEEN :checkIn AND :checkOut - INTERVAL 1 DAY
ORDER BY date ASC
FOR UPDATE;
```

- `available_count > 0` 조건을 SQL에 넣지 않는다. 3박 예약 시 중간 날짜가 매진이면 해당 행이 결과에서 누락되어 `"3박인데 2행만 반환"`되는 문제가 발생하기 때문이다.
- 대신 전체 날짜의 행을 락 건 뒤, 애플리케이션에서 조회된 행 수가 요청 숙박일 수와 일치하는지, 각 행의 가용 재고가 0보다 큰지를 검증한다.

---

## 대안

- Option A: 낙관적 락 (`@Version`)
  - 장점
    - 락 대기가 없으므로 충돌이 적은 평시에 처리량이 높다.
    - 커넥션을 장시간 점유하지 않는다.
  - 단점
    - 충돌 시 `OptimisticLockException`이 발생하고 애플리케이션 레벨에서 재시도해야 한다.
    - 경합이 높을수록 재시도 횟수가 폭발적으로 증가한다.
    - 멀티 나이트 예약 시 날짜 수에 비례하여 연쇄 충돌 위험 증가

- Option B: 분산 락 (Redis Redisson)
  - 장점
    - DB 커넥션과 독립적이다. 멀티 인스턴스 환경에서도 동일하게 동작한다.
  - 단점
    - Redis라는 추가 인프라가 필요하다.
    - 락 획득 실패 시 재시도 전략이 필요하다.

---

## 기대 결과
- DB가 락 순서를 보장하므로 애플리케이션 레벨의 재시도 로직이 불필요하다.
- 재고 초과 예약(overbooking)이 DB 레벨에서 원천 차단된다.
- 멀티 나이트 예약에서 `ORDER BY date ASC` 락 획득 순서로 데드락이 방지된다.

## 고민 기록 
- 커넥션 점유 시간 증가: 비관적 락은 트랜잭션이 락을 보유하는 동안 해당 레코드에 대한 다른 트랜잭션이 대기하게 만든다. 트랜잭션 처리 시간이 길어지면 커넥션 풀 고갈로 이어질 수 있다.
- 그래서 `innodb_lock_wait_timeout`을 적절히 설정하여(기본 50초는 너무 길다) 장시간 대기를 방지한다.
- 락 범위를 최소화하여 예약 트랜잭션 내에서 외부 API 호출이나 무거운 연산을 수행하지 않도록 한다.
- 1차 애플리케이션에서 CAS 처리를 하게되면, 정책을 어떻게 잡아야하는지에 대한 고민이 있었다.
  - 하지만, 개인적으로는 overSelling이 발생하지 않는 것을 최우선으로 생각하고 결정하였다.
