# ADR-002: 비관적 락 기반 예약 동시성 처리

## 상태

Accepted (2026-03-28)

---

## 배경

OTA 플랫폼에서 예약 생성은 반드시 재고 차감과 원자적으로 처리되어야 한다. 특히 다음 시나리오가 핵심 문제다.

- 마지막 1개 객실 시나리오: 동일 날짜, 동일 객실 유형에 대해 다수의 예약 요청이 동시에 도착하면, 재고가 0인 상태에서 중복 예약이 발생해서는 안 된다.
- 멀티 나이트 예약: 3박 예약의 경우 체크인~체크아웃 사이의 모든 날짜 재고를 동시에 차감해야 하며, 일부 날짜만 차감되고 나머지가 실패하는 부분 성공 상태가 존재해서는 안 된다.

재고 데이터(`inventory` 테이블)는 날짜별로 분리된 레코드로 관리된다. 예약 생성 트랜잭션에서 해당 레코드를 읽고 차감하는 과정이 동시에 실행될 때 정합성을 보장하는 전략을 선택해야 한다.

---

## 결정

MySQL `SELECT ... FOR UPDATE` (비관적 락) 을 채택한다. 단일 서버 환경에서는 Caffeine CAS(AtomicInteger)를 1차 필터로 두어 DB 도달 전에 매진 요청을 걸러낸다. (상세: [05-concurrency.md](../05-concurrency.md) 7.3절)

- 예약 생성 트랜잭션 시작 시 대상 날짜의 `inventory` 레코드 전체를 `SELECT ... FOR UPDATE`로 획득한다
- 락을 획득한 트랜잭션만 재고 수량을 읽고 차감할 수 있으며, 다른 트랜잭션은 락이 해제될 때까지 대기한다
- 데드락 방지를 위해 멀티 나이트 예약 시 항상 `date ASC` 순서로 락을 획득한다

```sql
SELECT * FROM inventory
WHERE room_type_id = :roomTypeId
  AND date BETWEEN :checkIn AND :checkOut - INTERVAL 1 DAY
ORDER BY date ASC
FOR UPDATE;
```

- `available_count > 0` 조건을 SQL에 넣지 않는다. 3박 예약 시 중간 날짜가 매진이면 해당 행이 결과에서 누락되어 "3박인데 2행만 반환"되는 문제가 발생하기 때문이다
- 대신 전체 날짜의 행을 락 건 뒤, 애플리케이션에서 조회된 행 수가 요청 숙박일 수와 일치하는지, 각 행의 가용 재고가 0보다 큰지를 검증한다

## 대안

### Option A: 낙관적 락 (`@Version`)

JPA의 `@Version` 컬럼을 사용하여 커밋 시점에 충돌을 감지하고 재시도하는 방식이다.

- 장점:
  - 락 대기가 없으므로 충돌이 적은 평시에 처리량이 높다
  - 커넥션을 장시간 점유하지 않는다
- 단점:
  - 충돌 시 `OptimisticLockException`이 발생하고 애플리케이션 레벨에서 재시도해야 한다
  - 경합이 높을수록 재시도 횟수가 폭발적으로 증가한다 (100 동시 요청, 재고 1개 시 최대 약 397회 UPDATE 발생)
  - 멀티 나이트 예약 시 날짜 수에 비례하여 연쇄 충돌 위험 증가
- 판정: 본 프로젝트의 핵심 시나리오가 "높은 경합"이므로 재시도 비용이 실질적 부담이 된다. 불채택.

### Option B: 분산 락 (Redis Redisson)

Redis를 이용하여 `roomTypeId:date` 키 단위로 분산 락을 획득하는 방식이다.

- 장점: DB 커넥션과 독립적이다. 멀티 인스턴스 환경에서도 동일하게 동작한다.
- 단점:
  - Redis라는 추가 인프라가 필요하다
  - 락 획득 실패 시 재시도 전략이 필요하다
  - Redis 장애 시 예약 불가 상태가 된다
  - 현재 요구사항에서 단일 DB로 충분한 동시성 제어가 가능함에도 불구하고 인프라 복잡도를 높인다
- 판정: 인프라 비용 대비 현재 시점에서 얻는 이득이 없다. 멀티 인스턴스 운영이 필요한 시점에 재검토. 불채택.

## 결과

### 긍정적 결과

- DB가 락 순서를 보장하므로 애플리케이션 레벨의 재시도 로직이 불필요하다.
- 재고 초과 예약(overbooking)이 DB 레벨에서 원천 차단된다.
- 멀티 나이트 예약에서 `ORDER BY date ASC` 락 획득 순서로 데드락이 방지된다.

### 부정적 결과 및 관리 방안

- 커넥션 점유 시간 증가: 락을 보유한 트랜잭션이 완료될 때까지 해당 `inventory` 레코드에 대한 다른 트랜잭션은 대기한다. 트랜잭션 처리 시간이 길어지면 커넥션 풀 고갈로 이어질 수 있다.
  - 관리 방안:
    - `innodb_lock_wait_timeout`을 적절히 설정하여(기본 50초는 너무 길다) 장시간 대기를 방지한다.
    - 예약 트랜잭션 내에서 외부 API 호출이나 무거운 연산을 수행하지 않도록 트랜잭션 범위를 최소화한다.
- 처리량 한계: 동일 재고에 대한 예약은 직렬화된다. 초당 수천 건의 동일 객실 예약 요청이 있는 극단적 상황에서는 병목이 될 수 있다.
  - 관리 방안:
    - 실제 OTA에서 동일 객실 유형 예약의 동시 경합은 제한적일 것 같다.
    - 처리량 한계에 도달하는 시점에 Redis 분산 락 전환을 검토한다.

## 고민 기록

처음에는 낙관적 락을 고려했다.

- 충돌이 적은 평시에는 성능이 좋기 때문이다.
- DB 커넥션을 장시간 점유하지 않고, 대부분의 요청이 첫 시도에 성공하는 상황이라면 낙관적 락이 더 효율적이다.

하지만 본 프로젝트의 높은 경합 시나리오에서의 동시성 처리 증명이라는 점을 고려하면, 낙관적 락은 적합하지 않다고 생각했다.

- 재시도 비용이 폭발하는 낙관적 락은 적합하지 않다.
- 100개의 동시 요청이 1개의 재고를 두고 경쟁할 때, 낙관적 락은 99개의 실패 트랜잭션이 각자 재시도를 반복하는 혼잡 상태를 만든다.
- 비관적 락은 이 상황을 100개의 순차 처리로 바꾸어 결과를 예측 가능하게 만든다.

## 후속 조치

- [ ] `innodb_lock_wait_timeout` 설정값 결정 및 문서화 (권장: 10초 ~ 50초 이하 적절한 값)
- [ ] 락 대기 시간 메트릭 수집 (`information_schema.innodb_lock_waits` 모니터링)
- [ ] 커넥션 풀 사이징 모니터링 (HikariCP `maximumPoolSize` 튜닝 기준 수립)
- [ ] 부하 테스트를 통한 비관적 락 vs 낙관적 락 실측 비교 (JMeter 또는 Gatling)
- [ ] 트래픽 증가 시 Redis 분산 락 전환 시점 기준 정의
