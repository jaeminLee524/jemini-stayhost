# ADR-003: Caffeine 로컬 캐시 + 하위 단위 캐시 키 전략

## 상태

Accepted (2026-03-28)

---

## 배경

검색 API는 OTA 플랫폼에서 많은 요청을 처리하는 엔드포인트다.

- 사용자가 지역, 날짜, 인원수를 입력하면 조건에 맞는 숙소 목록과 요금을 반환해야 한다
- 요금 계산은 날짜별 `rate_policy` 레코드를 읽고, 채널 마진을 적용하고, 가용 재고를 확인하는 여러 DB 조회를 포함한다
- 캐시 없이는 동시 다수 검색 요청이 DB에 그대로 전달되어 응답 지연과 DB 부하가 증가한다

캐시 전략을 설계할 때 캐시 키의 단위가 핵심 결정 사항이다. 키가 너무 크면 조합 폭발로 히트율이 떨어지고, 너무 작으면 캐시 구성에 필요한 조회 횟수가 늘어난다.

---

## 결정

Caffeine 로컬 인메모리 캐시를 사용하며, 캐시 키 단위를 `property`, `roomType`, `ratePolicy` 단위로 분리한다.

### 캐시 구성

```
property:{propertyId}          → Property 기본 정보 (이름, 주소, 유형 등)
roomType:{roomTypeId}          → RoomType 정보 (이름, 최대 인원, 어메니티)
ratePolicy:{roomTypeId}:{date} → 해당 날짜의 요금 정책
```

검색 요청이 들어오면 조건에 맞는 `propertyId` 목록을 DB에서 조회한 후, 각 숙소의 상세 정보와 요금은 위 캐시에서 조립한다.

### Caffeine 설정

```java
Caffeine.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(Duration.ofMinutes(5))  // TTL: 안전망
    .recordStats()  // 히트율 모니터링
    .build();
```

---

## 대안

### Option A: Redis 글로벌 캐시

Redis를 분산 캐시로 사용하는 방식이다.

- 장점:
  - 멀티 인스턴스 환경에서 모든 인스턴스가 동일한 캐시 데이터를 공유한다
  - Extranet에서 요금 변경 후 즉시 모든 인스턴스의 캐시가 갱신되므로 정합성이 높다
- 단점:
  - 추가 인프라(Redis 서버)가 필요하다
  - 캐시 조회 자체가 네트워크 I/O를 포함한다
  - 로컬 캐시 대비 레이턴시가 높다
- 판정: 단일 인스턴스 환경에서 인프라 비용 대비 효용이 낮다. 멀티 인스턴스 운영 시 전환 대상. 현재는 불채택.

### Option B: 검색 결과 통째로 캐시

`searchResult:{region}:{checkIn}:{checkOut}:{guests}:{page}` 형태로 검색 결과 전체를 캐시하는 방식이다.

- 장점: 동일 조건 재검색 시 DB 조회가 전혀 없다.
- 단점: 캐시 키의 차원이 너무 많다.

조합 폭발 분석:

```
region:       전국 시/도 17개
checkIn:      연간 365일 (6개월 예약 가능 기준 ~180일)
checkOut:     checkIn 기준 1~14박 → 평균 7가지
guests:       1~10인 → 10가지
page:         1~N

총 조합 수 = 17 × 180 × 7 × 10 × N ≈ 수백만 ~ 수천만
```

- 실사용 환경에서 동일한 `(region, checkIn, checkOut, guests, page)` 조합이 반복될 확률은 극히 낮다
- 캐시가 가득 차도 히트율이 1% 미만일 수 있으며, 메모리만 낭비된다
- 숙소 하나의 요금이 변경되었을 때 해당 숙소를 포함하는 모든 검색 결과 캐시를 무효화해야 하므로 무효화 범위 특정이 불가능하다

판정: 히트율 극저, 무효화 불가로 불채택.

## 결과

### 긍정적 결과

- 요금 조회 DB 부하가 캐시 히트율에 비례하여 감소한다
  - `ratePolicy:{roomTypeId}:{date}` 단위로 히트율이 높다
  - 예를 들어 서울 숙소 A의 3월 30일 요금은 검색하는 모든 사용자가 동일한 캐시 엔트리를 공유한다
- 캐시 무효화가 정밀하다
  - 숙소 A의 요금이 변경되면 `ratePolicy:{A의 roomTypeId}:*` 키만 무효화하면 된다
  - 검색 결과 전체를 무효화할 필요가 없다
- 추가 인프라 없이 단일 Spring Boot 애플리케이션 내에서 동작한다

### 부정적 결과 및 관리 방안

- 멀티 인스턴스 시 캐시 불일치: 인스턴스 A에서 요금이 변경되어 캐시를 무효화해도, 인스턴스 B의 로컬 캐시는 TTL이 만료될 때까지 구버전 데이터를 반환한다.
  - 관리 방안:
    - Spring ApplicationEvent 기반 무효화 이벤트(ADR-005)로 동일 인스턴스 내 정합성을 유지한다
    - TTL(5분)을 안전망으로 사용한다
    - 멀티 인스턴스 운영 시 Redis Pub/Sub 전환
- JVM 힙 사용량 증가: 캐시 데이터가 JVM 힙 메모리를 점유한다.
  - 관리 방안: `maximumSize` 설정으로 상한을 제한하고, Caffeine의 `recordStats()`로 히트율과 캐시 크기를 모니터링한다.

## 고민 기록

글로벌 캐시(Redis)로 Extranet과 정합성을 유지하는 것이 이상적이다.

- 파트너가 요금을 변경하면 모든 인스턴스에서 즉시 반영되기 때문이다.

그러나 현재 환경에서는 로컬 캐시를 채택한다.

- 현재는 멀티 인스턴스 환경은 오버 스펙이라고 판단했다.
- Redis 인프라를 추가하는 것은 본 프로젝트의 핵심인 도메인 로직과 동시성 처리 증명에 집중할 시간을 분산시킨다.
- 로컬 캐시 + ApplicationEvent 무효화로 현재 요구사항을 충분히 충족할 수 있다.
- Redis 전환 경로도 명확하게 설계되어 있다.

## 후속 조치

- [ ] Caffeine `recordStats()` 기반 캐시 히트율 로깅 구현
- [ ] `ratePolicy` TTL을 요금 변경 빈도에 맞게 튜닝 (현재 5분)
- [ ] 멀티 인스턴스 운영 시 Redis Pub/Sub 기반 캐시 무효화 전환 (ADR-005와 연계)
- [ ] 캐시 웜업 전략 검토 (서버 재시작 직후 콜드 스타트 완화)
