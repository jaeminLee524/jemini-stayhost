# ADR-001: 모놀리식 + 패키지 분리 아키텍처 채택

## 상태

Accepted (2026-03-28)

---

## 배경

이 플랫폼은 Partner, Property, Booking, Search, Channel Manager, Supplier 등 최소 6개의 명확히 구분되는 Bounded Context를 포함하는 OTA 도메인이다.

아키텍처를 결정할 때 두 가지 상충하는 목표가 있었다.

1. 구현 완성도: 동시성 처리, 캐시 전략, 예약 플로우가 실제로 동작하는 코드로 증명되어야 한다.
2. 아키텍처 깊이: 도메인 경계가 명확하게 표현되어야 하고, 향후 확장 가능한 구조여야 한다.

---

## 결정

단일 Spring Boot 애플리케이션 + 도메인별 패키지 분리 구조를 채택한다.

패키지 구조는 다음과 같이 Bounded Context를 반영한다.

```
com.jemini.stayhost
├── partner/          # Partner Context
├── property/         # Property Context (숙소, 객실, 요금, 재고)
├── booking/          # Booking Context (예약, 취소)
├── search/           # Search Context (검색, 캐시)
├── customer/         # Customer Context (회원, 인증)
├── channel/          # Channel Manager Context (DESIGN-ONLY)
└── supplier/         # Supplier Context (DESIGN-ONLY, 매핑 테이블은 BUILD)
```

각 Context 내부는 `domain`, `application`, `infrastructure`, `api` 레이어로 구성하여 헥사고날 아키텍처의 의존성 방향을 유지한다.

---

## 대안

### Option B: 멀티 모듈 Gradle

Gradle 멀티 모듈로 각 Context를 물리적으로 분리하는 방식이다.

- 장점:
  - 컴파일 타임에 모듈 간 의존성을 강제할 수 있다
  - 빌드 격리로 변경 범위가 명확해진다
- 단점:
  - `settings.gradle.kts`, 각 모듈의 `build.gradle.kts`, 공유 DTO 모듈 설정 등 초기 설정 비용이 상당하다
  - 본 프로젝트의 목적에서 이 비용은 실질적인 비즈니스 로직 구현 시간을 잠식한다
- 판정: 아키텍처적으로는 올바른 방향이지만, 시간적 제약 위배 위험으로 불채택.

### Option C: MSA (마이크로서비스)

각 Context를 독립적인 서비스로 분리하는 방식이다.

- 장점: 독립 배포, 독립 스케일링, 장애 격리
- 단점:
  - 서비스 간 통신(REST/gRPC), 분산 트랜잭션(Saga/2PC), 서비스 디스커버리, API Gateway 등 인프라 복잡도가 수십 배 증가한다
  - 도메인 로직보다 인프라 구성에 더 많은 시간을 쓰게 된다
- 판정: 현재 고려하는 범위를 명백히 초과하며, 핵심 비즈니스 로직 검증이 불가능해지므로 불채택.

---

## 결과

### 긍정적 결과

- 단일 트랜잭션 내에서 여러 Context에 걸친 작업을 자연스럽게 처리할 수 있다
  - 예약 생성 시 재고 차감과 예약 레코드 생성이 하나의 트랜잭션으로 묶인다
- JPA 연관 관계 탐색이 단순하다
  - 서비스 간 HTTP 호출 없이 직접 레포지토리를 사용할 수 있다
- 짧은 기간 내 동시성 처리, 캐시 전략, 예약 플로우가 모두 동작하는 코드로 완성 가능하다

### 부정적 결과 및 관리 방안

- 패키지 간 직접 참조 가능: 단일 JVM 위에 있으므로 `booking` 패키지에서 `property` 패키지의 내부 클래스를 직접 참조하는 것을 언어 수준에서 막을 수 없다.
  - 관리 방안: Context 간 참조는 반드시 `application` 레이어의 인터페이스 또는 공개 API 클래스를 통해서만 허용한다는 규칙을 코드 리뷰로 강제한다
    - 향후 ArchUnit 테스트로 자동 검증할 수 있다
- 향후 분리 비용: 서비스가 성장하여 MSA로 전환할 경우, 암묵적 결합이 존재할 수 있다.
  - 관리 방안: 초기부터 Context 경계를 엄격히 유지하고, 교차 참조 지점을 최소화하여 분리 비용을 낮춘다.

## 트레이드오프 요약

| 항목 | 모놀리식 + 패키지 분리 | 멀티 모듈 | MSA |
|------|----------------------|-----------|-----|
| 구현 속도 | 빠름 | 중간 | 느림 |
| 모듈 격리 강도 | 약함 (컨벤션 의존) | 강함 (컴파일 타임) | 매우 강함 (배포 단위) |
| 트랜잭션 관리 | 단순 | 단순 | 복잡 (분산 트랜잭션) |

핵심 트레이드오프: 구현 속도와 완성도를 위해 모듈 격리 강도를 포기했다. 이 선택은 제약이 해소되고 팀이 성장하면 재검토해야 한다.

## 후속 조치

- [ ] ArchUnit 의존성 규칙 테스트 추가 (`booking` → `property` 직접 참조 금지 등)
- [ ] Context 간 참조 지점 문서화 (Anti-Corruption Layer 후보 식별)
- [ ] 서비스 성장 시 멀티 모듈 전환 기준 정의 (팀 규모, 배포 독립성 필요 여부)
