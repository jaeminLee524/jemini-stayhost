# AI 활용 기록

## 활용 방침
- AI를 설계 초안 작성, 코드 리뷰, 트레이드오프 분석에 활용한다.
- AI가 생성한 내용을 그대로 사용하지 않고, 본인의 판단을 더해 수정/보완한다.
- 모든 활용 내역을 기록한다.

---

## Day 1 - 설계 계획 수립 및 문서 작성

### 설계 계획 프로세스

- AI(Claude)의 RALPLAN 컨센서스 워크플로우를 활용하여 초기 설계 계획을 수립했다. RALPLAN은 Planner(계획) → Architect(아키텍처 리뷰) → Critic(품질 평가) 3단계 합의 루프로, 설계안에 대해 다각도로 검증하는 방식이다.
- 이 과정은 단순히 AI가 계획을 내놓고 그대로 따른 것이 아니다.
- 초안(v1~v3)에서 출발하여 과제 요구사항을 직접 분석한 후, 기술 스택 변경, 채널 매니저 개념 추가, 문서화 우선 전략, Supplier 배치 설계 등 핵심 방향을 직접 지정하고 AI에게 재설계를 요청했다. 
  - 최종 v5에서 Architect와 Critic 모두 APPROVE를 받은 뒤에도 당근마켓 아키텍처를 참고한 DIP 패키지 구조, 자주 사용하던 ApiBaseResponse/Swagger 패턴 도입 등 추가 개선을 직접 결정했다.
- 요약하면, AI는 초안 생성과 트레이드오프 분석 도구로 활용했고, 최종 설계 결정은 모두 본인의 판단으로 이루어졌다.

### 활용 내역

| 작업 | AI 활용 방식 | 본인 판단/수정                                                                                                                                   |
|------|-------------|--------------------------------------------------------------------------------------------------------------------------------------------|

| 동시성 전략 | 낙관적 락 vs 비관적 락 비교 분석 요청 | 초기에는 낙관적 락을 고려했으나, 100 동시 요청 시나리오에서 재시도 비용을 계산하여 비관적 락으로 전환 결정                                                                             |
| 캐시 전략 | 검색 결과 캐시 vs 하위 단위 캐시 비교 요청 | 검색 파라미터 조합 폭발(43만 가지)을 직접 계산하여 검색 결과 캐시 폐기 결정. 글로벌 캐시 대신 로컬 캐시 선택은 본인 판단                                                                   |
| ERD 설계 | 테이블 구조 초안 생성 | rate/inventory 분리, reserved_count 방식 선택, reservation_daily_rate 스냅샷 설계는 직접 판단                                                              |

| 패키지 아키텍처 | 당근마켓 헥사고날 아키텍처 구조 분석 요청 | 풀 헥사고날은 본 프로젝트 목적에 과하다고 판단. DIP 원칙만 추출하여 모놀리식에 적용하는 방향을 직접 결정 (domain → infrastructure 역전)                                                 |
