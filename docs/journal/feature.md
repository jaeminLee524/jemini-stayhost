# 기능 별 고민
각 기능을 설계 및 개발하기 앞서 고려해야 할 사항들을 정리한 문서입니다.

- 전제
  - 전제를 정의해야지, 기능 구현의 범위를 산정할 수 있을 것 같았다.
  - 우선, 해당 프로젝트는 단일 인스턴스 환경에서 운영된다고 가정한다.
  
## 1. 숙소 검색 
### 1-1 대규모 요금 조회 요청에 대한 설계를 어떻게 할 것인가?
- 요금 조회는 숙소 검색과 함께 가장 빈번하게 발생하는 작업이므로, DB 부하를 줄이는 방안을 고민해야 한다.

### 1-2 자료조사
- 현재 여기어때, 야놀자 등 국내 숙박 예약 플랫폼에서 요금 조회에 대한 검색 조건이 어떻게 되는지 조사한다.

### 1-3 문제 정의
- 캐시 미적용시, 현재 구조상 아래의 극단적인 DB 쿼리가 발생할 수 있다.
- 숙소 20개 * 객실 3개 * 30일 = 1,800 rate(요금) 조회
- 100명이면 초당 180,000 rate 조회가 발생한다.

### 1-4 해결 방안
 - 해당 프로젝트에서는 캐시 갱신의 주 수단을 이벤트 기반의 eviction 으로 설계했다.
 - TTL 은 안전망이다. 이벤트 누락 대비하여 5분으로 설정했다.
 - 캐시 구성은 아래와 같다.

| 캐시        | 키                   | TTL |
|-----------|---------------------|-----|
| property  | {propertyId}        | 3분  |
| roomTypes | {propertyId}        | 3분  |
| rate      | {roomTypeId}:{date} | 3분  |

### 1-5 검색 플로우 & 동시 요청 효과 & 캐시 무효화 전략

검색 플로우 시나리오

- Phase 1: 검색 목록 (제주, 4/9-4/17)
  ```
  1. 커버링 인덱스 -> 숙소 ID 20개(DB, 캐시 미적용)
  2. RoomType 조회 (캐시 히트 or DB)
  3. Rate 조회 (roomType 별 warm-up)
            - Cold: 20개 숙소 * 3개 객실 유형 = 60개 DB 조회 후 캐시 적재
            - Warm: 캐시에서 바로 조회(DB 조회 없음)
  4. 숙소별 최저가 계산 후 응답
  ```
  
- Phase 2: 숙소 상세 클릭
  ```
  1. Property 조회 (캐시 히트)
  2. RoomType 조회 (캐시 히트)
  3. Rate 조회 -> Phase 1에서 이미 캐시가 적재되어 있으므로 DB 조회 없음
  4. Inventory 조회 (DB 조회, 캐시 미적용 대상)
  5. 객실별 날짜별 요금 + 예약 가능 여부 계산 후 응답
  ```
          

동시 요청 효과
  
  - A 사용자 (Cold) :Rate 조회 60회 → Inventory DB 1회 -> 캐시 적재
  - B 사용자 (Warm) :Rate 조회 0회 → Inventory DB 1회
  - C 사용자 (Warm) :Rate 조회 0회 → Inventory DB 1회

캐시 무효화 전략

  - 파트너 요금 수정 -> RateUpdatedEvent    -> rate:{roomTypeId}:{date} 즉시 evict                      
  - 파트너 숙소 수정 -> PropertyUpdatedEvent -> property:{id} 즉시 evict                                
  - 파트너 객실 수정 -> RoomTypeUpdatedEvent -> roomTypes:{propertyId} 즉시 evict                       
  - 파트너 재고 변경 -> InventoryChangedEvent -> 캐시 대상 아님 (항상 DB)
  - 예약 생성/취소 -> 재고 변동 -> 캐시 대상 아님 (항상 DB)

## 2. 예약(동시성)
### 2-1 각 동시성 특징

  - 낙관적 락, 비관적 락, 분산 락 등 여러 가지 동시성 제어 방법이 있다. 각각의 장단점을 분석하여 적절한 방법을 선택해야 한다.
  - 적은 트래픽 내에서는 낙관적 락이 효율적일 수 있지만, 트래픽이 증가할 경우 충돌이 빈번해질 수 있다.
  - 반면 비관적 락은 충돌이 많아도 안정적으로 처리할 수 있지만, DB 커넥션을 장시간 점유할 수 있다.
  - 전제를 단일 인스턴스 환경으로 했으므로, 분산 락은 불필요한 복잡성을 추가할 수 있다.

### 2-2 동시성 제어 방법 선택
  - 이번 프로젝트에서는 미리 메모리(향후 캐시) 재고 정보를 warm-up 해두고, 예약 트랜잭션에서는 DB에서 재고를 직접 조회하여 비관적 락을 적용한다.
  - 미리 메모리에 재고 정보를 올려두어 over booking 가능성을 줄이고, 불필요한 트래픽이 DB로 가지 않도록 한다.

### 2-3 예약 동시성 플로우
- 시나리오: 100명 사용자, 재고 10개
  ```
  1. 로컬에 객실유형:날짜 별 재고 정보를 warm-up 해둔다.
  2. 예약 요청이 들어오면, CAS 10명 성공 후 나머지 90명은 메모리 재고 부족으로 바로 실패한다.
  3. 비관적 락으로 DB에서 X-lock을 획득하여 재고를 차감한다.
    - 데드락 방지를 위해 Order By room_type, date ASC 순서로 락을 획득한다.
  4. 순차적으로 락을 획득한다.
  5. TOCTOU 재검증을 한다. 캐시와 DB 간 불일치를 방어하기 위함
  6. 재고차감 & 예약 확정
  ```

## 3. 테스트 전략

### 3-1 테스트 전략을 어떻게 구성할 것인가?
- 단위 테스트만으로는 동시성, 트랜잭션 경계, 이벤트 발행 등 인프라와 결합된 동작을 검증할 수 없다.
- 반대로 통합 테스트만으로는 피드백 속도가 느려 개발 생산성이 떨어진다.
- 어떤 범위를 어떤 레벨에서 검증할지 기준이 필요했다.

### 3-2 테스트 레벨 분류와 선택 기준

| 레벨 | 대상 | 도구 | DB |
|------|------|------|----|
| 단위 테스트 | 도메인 모델, 서비스 로직 | JUnit5 + Mockito | X (mock) |
| 통합 테스트 | API 전체 플로우, 동시성 | SpringBootTest + TestRestTemplate | O (실제 MySQL) |

### 3-3 의사결정: 통합 테스트에서 실제 DB를 사용한 이유

- 대안: H2 인메모리 DB 또는 mock Repository
  - 장점: 테스트 속도 빠름, 환경 구성 간단
  - 단점: SELECT FOR UPDATE 동작이 MySQL과 다름, 트랜잭션 격리 수준 차이, 실제 데드락 재현 불가
- 선택: Testcontainers MySQL
  - 장점: 운영 환경과 동일한 DB 엔진에서 검증, 비관적 락/데드락/트랜잭션 격리 수준 동작이 실제와 동일
  - 단점: 테스트 속도 느림 (컨테이너 기동 약 5~10초)
  - 이 프로젝트에서 동시성이 핵심이므로, 실제 DB에서 검증하는 것이 mock 대비 훨씬 신뢰도가 높다고 판단했다.

## 4. 이벤트 기반 아키텍처

### 4-1 왜 이벤트가 필요한가?
- 예약이 생성되면 후속 작업이 여러 개 발생한다(캐시 무효화, 채널 동기화, 알림 발송 등)
- 이 후속 작업들을 예약 서비스에 직접 넣으면 강결합이 발생한다.
- 이벤트로 분리하면 발행자와 구독자가 느슨한 결합을 달성한다.

### 4-2 메시징 시스템 선택

| 항목 | Spring ApplicationEvent | Kafka / RabbitMQ |
|------|------------------------|-----------------|
| 인프라 | 추가 없음 (JVM 내 동작) | 별도 브로커 서버 운영 |
| 트랜잭션 연계 | @TransactionalEventListener로 커밋 후 처리 보장 | Transactional Outbox Pattern 필요 |
| 메시지 영속성 | 없음 (처리 실패 시 유실) | 있음 (디스크 저장) |
| 멀티 인스턴스 전파 | 발행 인스턴스 내부에서만 | 모든 소비자에 전파 |

- 선택: Spring ApplicationEvent
- 이유: 단일 인스턴스 전제에서 Kafka는 과도한 복잡도. JVM 내 이벤트로 충분하다.

### 4-3 @EventListener vs @TransactionalEventListener

- @EventListener: 트랜잭션 내에서 이벤트 처리가 실행된다.
  - 문제: 예약 생성 이벤트 → 채널 동기화 실행 → 이후 트랜잭션 롤백 → 채널에는 잘못된 재고가 전달된 상태
- @TransactionalEventListener(phase = AFTER_COMMIT): DB 커밋이 완료된 후에만 처리한다.
  - 데이터가 확정된 후 부수 효과를 실행하므로 불일치 원천 방지
  - 트랜잭션 롤백 시 이벤트 자체가 발행되지 않음 (의도된 동작)

- 선택
  - @TransactionalEventListener(AFTER_COMMIT)
- 이유
  - 커밋 전 처리 시 롤백에 의한 불일치가 발생할 수 있고, 이를 보상하는 로직이 오히려 복잡하다.

### 4-4 이벤트 유실 허용 판단

- AFTER_COMMIT에서 처리 실패 시 이벤트가 재발행되지 않는다. 유실을 어디까지 허용할 것인가?

| 이벤트 유형 | 유실 허용 | 안전망 |
|------------|---------|--------|
| 캐시 무효화 (PropertyUpdatedEvent 등) | 허용 | TTL 3~5분이 2차 안전망 |
| 재고 캐시 동기화 (InventoryChangedEvent) | 허용 | DB 비관적 락이 최종 정합성 보장 |
| 채널 동기화 (ReservationCreatedEvent) | DESIGN-ONLY | 현재는 로그 기록만, 프로덕션에서는 재시도 큐 필요 |

- 현재 구현 범위에서는 캐시 무효화와 재고 캐시 동기화만 BUILD 대상이다.
- 두 경우 모두 TTL 또는 DB 락이 2차 안전망 역할을 하므로 이벤트 유실이 치명적이지 않다.
- 메시지 유실이 허용되지 않는 핵심 비즈니스(결제 등)는 Transactional Outbox Pattern이 필요하지만, 현재 범위에서는 불필요하다고 판단했다.

### 4-5 멀티 인스턴스 전환 시 변경 경로

- ApplicationEvent의 한계: 발행한 인스턴스에서만 이벤트가 처리된다.
- 멀티 인스턴스로 전환하면 인스턴스 A에서 발행한 캐시 무효화 이벤트가 인스턴스 B에는 전달되지 않는다.
- 단계적 전환 경로를 설계해두었다.
  ```
  Phase 1 (현재): ApplicationEvent — 단일 인스턴스
  Phase 2: ApplicationEvent + Redis Pub/Sub — 캐시 무효화만 글로벌 전파
  Phase 3: Kafka — 이벤트 영속성 + 외부 시스템 연동 필요 시
  ```
- 서비스 로직(캐시 무효화, 채널 동기화 처리)은 변경 없이 재사용된다. 발행/구독 메커니즘만 교체하면 되도록 설계했다.

## 6. 인증 설계

### 6-1 인증 방식을 어떻게 선택할 것인가?
- 서버가 사용자의 인증 상태를 어떻게 관리할 것인지 결정해야 한다.
- 크게 세션 기반(Stateful)과 토큰 기반(Stateless) 두 가지가 있다.

### 6-2 세션 vs JWT

| 기준 | 세션 방식 | JWT 방식 |
|------|----------|---------|
| 서버 상태 | Stateful (세션 저장소 필요) | Stateless (토큰 자급) |
| 수평 확장 | Redis 등 공유 스토어 필요 | 별도 인프라 불필요 |
| 강제 로그아웃 | 세션 삭제로 즉시 가능 | 토큰 만료까지 대기 (블랙리스트 필요) |
| 구현 비용 | Spring Session + Redis 설정 | jjwt 의존성 + 필터 1개 |

- 선택: JWT
- 이유
  - Redis 세션 스토어 없이도 수평 확장이 가능한 구조
  - 이 프로젝트의 핵심은 인증 인프라가 아닌 숙소 예약 도메인이므로, 인증은 가볍게 유지
  - 강제 로그아웃 불가 트레이드오프는 Access Token TTL 30분으로 완화

### 6-3 파트너/고객 인증 체계 분리 vs 통합

- 초기에 하나의 로그인 API로 고객과 파트너를 모두 처리하는 방안을 검토했다.
  - 이메일/패스워드 기반이라는 공통점이 있기 때문
- 그러나 분리를 결정한 근거는 다음과 같다.
  1. 엔티티가 다르다: 고객은 User, 파트너는 Partner (사업자번호, 계좌 등 포함)
  2. JWT claims가 다르다: 파트너 토큰에는 partnerId가 포함되어야 숙소 소유권 검증이 가능
  3. URL 패턴이 다르다: 고객은 /api/, 파트너는 /api/extranet/
- 통합했을 때의 이점(코드 중복 감소)보다 분리했을 때의 이점(명확한 책임, 독립적 확장, 보안 오류 방지)이 크다.

### 6-4 @AuthenticationPrincipal 대신 래퍼 타입 + ArgumentResolver

- Spring Security의 @AuthenticationPrincipal을 직접 사용하면 컨트롤러가 Spring Security 구현 세부사항에 의존한다.
  - 인증 방식이 JWT → OAuth2로 변경되면 모든 Controller를 수정해야 함
- 래퍼 타입(UserId, PartnerId) + HandlerMethodArgumentResolver로 분리하면
  - Controller는 인증 구현을 모른 채 필요한 값만 받는다
  - 인증 방식이 변경되어도 Resolver만 수정하면 Controller는 그대로

- 선택: 래퍼 타입 + ArgumentResolver
  ```
  Controller                  Resolver                    Security
  UserId userId  ←──────  UserIdResolver  ←──────  SecurityContextHolder
  PartnerId partnerId ←── PartnerIdResolver ←─────  JwtPrincipal
  ```
- 이유: 인증 구현과 비즈니스 로직의 결합도를 낮추기 위함. 향후 인증 방식 변경 시 변경 범위를 Resolver 1개로 한정할 수 있다.
